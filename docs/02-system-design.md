## 確定事項（v0.2 決定ログ）

本版で未決4点を確定。委ねられた2点（暗号執行先・再校正頻度）は推奨を採用根拠とともに明記する。

### D1. 暗号執行先 → bitbank（JFSA登録の国内取引所）を Phase 1-2 の標準
**推奨理由**：Phase 2の投入資本が〜50万円（最小検証）であり、この規模での最優先事項は「マイクロ構造アルファの最大化」ではなく、①法規制・税務の明快さ ②研究と執行の venue 一貫性（信号の転移リスク排除）。
- 日本居住者の実資金執行は **JFSA登録業者が法的に明快**。オフショアのデリバ（Bybit/OKX perp）は規制グレーゾーン・地理ブロック・税務複雑化のリスクがあり本フェーズでは見送り。
- bitbank は現物の板/約定を WebSocket/CCXT で取得でき、**ドルバー・基本的なインバランスバー・分数階差分・トリプルバリア等ツールキットの大半は適用可能**（VPIN級の高度な板分析のみ精度低下）。代替候補：GMO Coin。
- **研究データも執行 venue に揃える**（bitbank のデータで開発・検証）。Binance 等の公開データは無料・高品質だが、そこで検証した信号が国内現物に転移する保証がないため参考に留める。
- 将来スケール時のオフショア・デリバ採用は、税務・規制の専門家レビューを前提に再検討。

> 注：これは投資助言・法務税務助言ではない。実際の規制・税務の取り扱いは各自の状況で確認のこと。

### D2. Phase 2 運用資本 → 〜50万円（最小検証）
目的は配管・心理・スリッページ実測。コスト比率は高いが学習価値を優先。最小ロット・手数料を実値でモデル化し、live-vs-backtest 乖離の実測に集中。

### D3. 再校正頻度 → 月次再fit ＋ 週次監視 ＋ レジーム検知のイベント駆動オーバーライド
**決め方の原則**（頻度は勘でなく導出する）：
1. **ベット保有期間から導く**：再fit間隔はトリプルバリアの垂直バリア（保有ホライズン）の数倍を下限とする。日〜数日のベットなら月次が妥当。
2. **有効サンプル充足で上限を縛る**：非IIDのため、独自性重み付け後の「新規実効サンプル」が再fitを正当化する量に達してから再fitする。
3. **K予算（DSR）で罰則を意識**：再fitは実質的に新たな試行＝Kを増やしDSRを下げる。**頻繁な再fitは過学習**。ゆえに「速い適応」は定期再fitでなく**構造変化検知**で対応する [KB §8-3]。
4. **エンバーゴ連動**：エンバーゴ幅は最低でも垂直バリアのホライズン＋系列相関の減衰長をカバー。再fitはウォークフォワードで、ライブ期間のラベルと重なるデータでは再学習しない [KB §5.1]。
5. **スケジュールを事前登録**：再fit頻度はライブ前に固定（事前登録）。成績を見て頻度を反応的に変えると**メタ過学習**になる [DP10]。

→ 既定値：**月次再fit／週次監視／レジーム検知時のみ臨時再校正**。柱Cスリーブ（中長期）は四半期も可。

### D4. 監視ダッシュボード → 後回し（CLI/Jupyterで開始）
Phase 0-1 はログ/ノートブックで監視。UI構築は研究・検証の後。将来必要なら Next.js+Supabase(Postgres)/Vercel で軽量に追加。

### D5. Phase 2 前提と実装仕様（2026-06 確定）
**前提**：総資本 ~¥100万（既定配分：株式60万／TSMOM30万／証拠金バッファ10万、CLIで変更可）。
**ペーパー先行**（Phase 2a）→ 月次サイクルを2回連続無事故で回せたら小口実弾（2b）。
証券会社＝**楽天証券**（かぶミニ＝単元未満株、日経225マイクロ先物、楽天CFD・楽天FX、特定口座）。
公式 API は無いため、Phase 2b の執行は**生成された注文CSVからの月1回の手動発注**を既定とする
（マーケットスピードII RSS による半自動化は任意の将来課題）。

**実装仕様（小資金への写像＝`invest_system/production/orders.py`）**：
- 株式ロング脚＝単元未満株（かぶミニ・1株単位）。**寄付取引＝翌営業日始値の約定**で、
  検証の T+1 始値規約（DP17・§6.13）と一致する。
- 株式ショート脚＝個別信用売りは小資金で構成不能のため、**合計ショート想定元本を
  日経225マイクロ売り**に集約（執行近似。トラッキングエラーは月次照合の計測対象）。
- TSMOM＝225マイクロのみ枚数化、他資産はペーパー段階では想定元本のまま記帳
  （実弾移行時にブローカーのロット表で較正）。
- シグナル生成＝`examples/phase2_generate_orders.py`（月末実行・戦略パラメータは凍結）。

**頻度（事前登録）**：取引＝**月次**（月末決定→翌営業日寄付執行。月中は無取引、例外は
キルスイッチのみ）。検証＝**週次15分**（データ更新・DD監視・台帳照合）＋**月次**（約定照合・
スリッページ実測・live-vs-backtest レポート）＋**四半期**（実測コストでコストパネル再較正→
旗艦を更新後モデルで再判定）＋**12ヶ月**（Phase 3 ゲート判定）。

**監視水準（事前登録・合成 BT maxDD −5.4%／全部入り悲観 −11% から導出）**：
警報＝合成DD **−8%**／デリスク50%＝**−12%**／全停止＋ポストモーテム＝**−15%**
または実測スリッページがモデルの3倍超。成績を見た反応的なパラメータ変更は禁止（DP10）。

**要確認チェックリスト（口座開設時）**：かぶミニ取扱銘柄の PIT ユニバース上位300カバー率／
かぶミニ寄付取引のコスト体系／225マイクロの証拠金・手数料／楽天CFDの銘柄・ロット表・
金利調整額／特定口座とかぶミニ・先物口座の区分。

## 1. 目的・スコープ・前提

### 1.1 目的
López de Pradoの金融機械学習フレームワークを忠実に実装し、**統計的に偽でない（過学習でない）アルファ**を、個人〜少人数で**工業的規律をもって**抽出・運用するシステムを構築する。

### 1.2 ゴール（測定可能な定義）
- 全戦略を **DSR（デフレ・シャープレシオ）** で評価し、生Sharpeでの意思決定を一切しない [KB §5.3]
- 全試行を**自動記録**し、試行数 K を DSR に機械的に反映する研究監視を内蔵 [KB §2, §5.3]
- シグナル→ペーパー→小口実取引→拡大、を**定量ゲート**で前進する

### 1.3 スコープ
- **含む**：データ取込、情報バー、特徴量（因果フィルタ込）、ラベリング、モデリング、検証、ポートフォリオ構築、シグナル生成、執行/OMS（段階導入）、監視
- **当面含まない**：高頻度（マイクロ秒）執行、自前マッチングエンジン、オプションのディープ・ヘッジ（将来オプション [KB §8-5]）

### 1.4 前提・制約
- 運用主体は個人/2〜3名 → **属人作業を最小化し自動化で代替**
- 計算資源はローカル中心（必要時クラウドへ）
- 最大の敵は外部ではなく**自分自身のp-hacking** → 検証は「意志」でなく「コード」で強制する

## 3. 運用コンセプト（CONOPS）— ソロ・リーンラボ

### 3.1 「組立ライン」をソロでどう実現するか
López de Pradoの中核的組織原理は「シロ型クオンツ（全工程を1人で抱える徒労）を排し、専門分化した組立ラインにする」[KB §2]。だが運用主体は個人。**矛盾の解消＝人の専門分化を自動化パイプラインで代替する。**

| LdPの専門役割 | ソロ・リーンラボでの代替 |
|----------------|--------------------------|
| データ・キュレーション担当 | 自動取込・バー構築・分数階差分の**確定パイプライン**（一度作れば再実行のみ） |
| 特徴量エンジニア | 特徴量ライブラリ＋因果フィルタの**自動適用** |
| バックテスト検証担当（査読役） | **機械的バリデータ**：DSR/CPCV/パージングをコードで強制、人の裁量を排除 |
| ポートフォリオ構築担当 | NCO/ノイズ除去/ベットサイジングの**自動配分器** |
| 研究監視システム | **試行レジストリ**（改竄不能・自動K集計） |

### 3.2 ソロ最大のリスク＝自己過学習への構造的対策
1. **事前登録（Pre-registration）**：バックテスト実行前に「経済的/数理的仮説」をレジストリに記録 [KB §2 untainted]。事後の理論こじつけ（a posteriori）を禁止し、a priori理論に限定。
2. **改竄不能な試行ログ**：全試行を自動記録、削除不可。Kが自動加算されDSR閾値が自動的に上昇 [KB §5.3]。
3. **アンサンブルによる査読代替**：チーム議論ができない代わりに複数モデルを投票させ単一故障点を回避 [KB §6.2, DP9]。
4. **ホールドアウト封印**：最終検証データはフェーズ・ゲート時に1回だけ開封。

### 3.3 日次オペレーション（定常運用時のイメージ）
```
[日次バッチ]
  1. データ取込・バー更新・特徴量再計算
  2. モデル推論 → シグナル生成（方向×メタラベル確率→ベットサイズ）
  3. ポートフォリオ最適化（ノイズ除去共分散→NCO）
  4. リスクチェック（限度・ドローダウン・レジーム）
  5. 執行（フェーズに応じ：ペーパー / 小口実 / 本番）
  6. 約定照合・実績記録・live-vs-backtest乖離監視
[週次/月次]
  - 再校正（エンバーゴ付き）、DSR再評価、減衰戦略の退役
```

## 4. 対象資産の選定と推奨

「暗号資産・日本株が候補、最もうまく動く対象を選ぶ余地あり」との回答を受け、LdPツールキットへの適合度で評価する。

### 4.1 適合度比較

| 観点 | 暗号資産 | 日本株式 |
|------|----------|----------|
| ティック/板データ入手 | ◎ 無料・高品質（CCXT/取引所WS） | △ J-Quantsは日次中心、ティックは限定/有料 |
| 執行容易性 | ◎ 24/7・分割可・API平易 | ○ 証券API（立会時間・空売り規制・貸株コスト） |
| マイクロ構造ツール（ドル/インバランスバー・VPIN [KB §3.2]） | ◎ 起点として最適 | △ リテールはティック取得が壁 |
| 因果ファクター文献・ファンダ（柱C [KB §7]） | △ 薄い | ◎ クロスセクション・財務データ豊富 |
| 分数階差分 [KB §3.1] | ○ 価格系列で適用可 | ○ 同左 |
| 非定常性・レジーム | △ 厳しい（操作・急変） | ○ 相対的に穏当 |
| 資本障壁 | ◎ 低い | ○ 中 |
| End-to-Endを最速で一巡 | ◎ | △ |

### 4.2 推奨：二刀流を「資産でも」二刀流にする
選択された戦略が**柱B×柱Cの統合**である以上、**資産も役割分担させる**のが最も自然：

- **スリーブA（柱B主）＝暗号資産**：ML予測パイプライン＋マイクロ構造の**フルツールキットを最速でEnd-to-End検証**し、実取引到達を早める。LdP手法の自然な適合度が最も高い。
- **スリーブB（柱C主）＝日本株式**：因果ファクターのクロスセクション帰属。財務データを交絡因子として扱い、コライダーバイアス回避 [KB §7.3]。インフラ実証後にシグナル→実取引で後続投入。

> **立ち上げ順**：Phase 1〜2は**暗号資産で一巡**（執行・データの障壁が最低）。Phase 3で**日本株の柱Cスリーブ**を追加。コアは§6で**資産非依存**に設計し、両者を同一パイプラインに載せる。

### 4.3 「最もうまく動く対象」への回答
LdPツールキットはマイクロ構造起点のため、**自然な適合度は暗号資産/先物が最高**。日本株は**因果ファクター帰属**で差別化価値を出す。よって「どちらか一方」ではなく、**コアを共通化した二スリーブ**が、選択された統合戦略に対する最適解。

```
            └───────────────┬───────────────────────┘
                            ▼
        ┌────────────────────────────────────────┐
        │ 統合層：ノイズ除去共分散→NCO→ベットサイジング  │
        │ （二スリーブのシグナルをアンサンブル合成）       │
        └───────────────────┬────────────────────┘
                            ▼
                  シグナル → 執行/OMS（段階導入）
```

### 5.1 柱Cが柱Bを強化する2経路
1. **特徴量フィルタ**：因果探索（PC/LiNGAM）で、スリーブAの特徴量から**合流点（コライダー）を除去**し、偽相関由来の特徴量を排除 [KB §7.2-7.3]。予測の堅牢性が上がる。
2. **アトリビューション**：スリーブBはリターンを**リスク源泉に帰属**し、ポートフォリオのリスク説明とガバナンスを担保 [KB §7.1]。

### 5.2 柱D：時系列・統計的裁定（mean reversion / stat-arb）【拡張・Ernie Chan 補完】

予測（柱B）でも risk-premium 帰属（柱C）でもない**第3の独立アルファ源**＝スプレッドの平均回帰。資産非依存・**市場中立（ダラーニュートラル）**で、既存スリーブと**相関が低い**ため、ポートフォリオの breadth（IR≈IC×√breadth）を直接押し上げる。

- **構築**：共和分でスプレッドを作り（CADF／Johansen、ヘッジ比は AsOf）、半減期で保有ホライズンを導出、z-score で逆張り。動的ヘッジは Kalman（オンライン＝先読みなし）[KB §11]。
- **検証**：柱B/C と同一ファネル（`Strategy` 契約 → `judge_grid` → DSR/試行レジストリ）。**ペア探索は全候補を試行計上**して DSR でデフレート（DP13）。
- **既存資産で即検証可**：取得済み日足全銘柄ミラー＋業種マスタ（S33）で、追加データ取得なしに業種内ペアを探索できる。
- **既存の萌芽**：`research/strategy.py` の `PairsStrategy`（素朴版）を、共和分ゲート＋動的ヘッジに昇格させて発展させる。

> コアは引き続き**資産非依存**（バー／分数階差分／独自性／CPCV／DSR）。柱D は新スリーブだが、共通コアと検証ファネルを**再利用する追加**であって、L1-L13 の骨格は変えない。

## 7. コンポーネント詳細

### L1 取込・ストレージ
- 暗号：**bitbank**（JFSA登録）のCCXT/WebSocketでtrades/depth、レート制限・再接続・欠損補完。研究データも執行と同一venueに揃え信号の転移リスクを排除（D1）。
- 日本株：J-Quants API（日次OHLCV・財務・指数構成）。**生存者バイアス対策**として時点別の指数構成を保持（上場廃止銘柄を当時の構成で再現）[a_data_science_solution の市場中立債券戦略と同じ思想]。
- ストレージ：特徴量は列指向（Parquet/DuckDB）、試行レジストリ・シグナルはPostgres（Supabaseで代替可）。

### L2 バー構築 [KB §3.2]
- ドルバー標準。情報優位検知にドル**インバランス**バー（オーダーフロー不均衡）。暗号は板/約定から構築可。日本株（日次中心）はドルボリューム近似バー＋将来ティック拡張。

### L3 分数階差分 [KB §3.1]
- ADF検定を通す**最小d**を動的探索し、原系列との相関を最大化。資産ごとにd校正（暗号/先物はd<0.5目安）。**d=1リターンは禁止**。

### L4 特徴量＋因果フィルタ [KB §7, §9]
- 特徴量：分数階差分系列、ボラ、マイクロ構造（VPIN等）、（日本株）財務・バリュー/クオリティ/モメンタム。
- **因果フィルタ**：候補特徴量に対しPC/LiNGAMで上流（交絡）/下流（合流点）を判定 → コライダー除去。ドメイン知識でDAGを先行定義し照合。
- 特徴量重要度：MDI/MDA/SFI。**重要度→理論構築→検証**の順（DP11）。

### L4+ 時系列ミーンリバージョン・ツールキット（柱D拡張）[KB §11]
- **検定**：半減期(OU)・Hurst・分散比・ADF/CADF で「平均回帰するか」をゲート。半減期が保有ホライズン/垂直バリア/再校正を導出（DP14）。
- **共和分/ヘッジ**：2資産は OLS/TLS、3本以上は Johansen 固有ベクトル。ヘッジ比は **AsOf 推定**（先読み厳禁・DP12）。
- **動的ヘッジ**：Kalman でヘッジ比・平均をオンライン更新（PIT 整合）。
- **戦略化**：`research/strategy.py` の `Strategy` 契約として薄く実装（`CointegratedPairs`/`JohansenBasket`/`LinearMeanReversion`）。既存 `PairsStrategy`（素朴版）の昇格を含む。**ペア探索は全候補を試行計上**（DP13）。
- 実装は `statsmodels`（adfuller/coint/coint_johansen）＋自前 Kalman。新規の重い依存は増やさない。

### L5 ラベリング [KB §4.1]
- トリプルバリア（利確/損切/時間）。バリア幅は実現ボラでスケール。パス依存を保持。

### L6 サンプル独自性 [KB §4.3]
- ラベル重複から独自性を算出、重み付け／逐次ブートストラップ。

### L7 モデリング [KB §4.2, §6.2]
- 一次モデル：方向（LightGBM等）。
- メタラベル：二次モデルが「賭けるか・サイズ」をPrecision最適化で判定。入力に一次予測の確率分布・不確実性を含める。
- アンサンブル：バギング/スタッキングで分散低減・単一故障点回避。

### L8 検証・ガバナンス [KB §5, §2]
- パージング＆エンバーゴ → CPCV（複数パス）→ Sharpe**分布**。
- DSR：K（試行数）・歪度・尖度・試行間分散で補正。False Strategy定理のE[max SR]と比較。
- **試行レジストリ**：全バックテストを自動記録（仮説・パラメータ・乱数種・結果・K）。**事前登録なきバックテストは拒否**。
- **執行現実性（DP15）**：値幅制限（ストップ高/安の引け張り付き）の執行不能＝前回ウェイトのキャリー、ショート貸株コスト（制度貸株料＋逆日歩バッファ）の期間按分控除、貸借銘柄カバレッジの診断（`equities/frictions.py` → `research/engine.py`）。日次の UL/LL フラグ・週次信用残の IssType を PIT で利用。
- **執行タイミングと状態依存コスト（DP17）**：同足終値執行（execution_lag=0）の先読みは、月次研究では `open_fill_backtest`（T+1 始値・open→open 実現）で再評価する。コストはスカラ bps に加えて銘柄×日付の bps パネル（`frictions.vol_scaled_cost_bps`＝実現σ連動）を受ける。

### L9 ポートフォリオ構築 [KB §6]
- 共分散をRMTでノイズ除去 → NCO（高相関ブロック分割）→ ベットサイジング。マークウィッツの逆行列直接法は不採用。
- **戦略全体のレバレッジ（DP16）**：実現ネットからフラクショナル・ケリー f = fraction×μ/σ²（複数戦略は F = fraction×C⁻¹M、C はノイズ除去後）で導出。既定 1/2、保守 1/4。満額禁止・月次再推定・ハード限度（最大DD/グロス上限）と併用（`portfolio/kelly.py`）。

### L11-L12 執行・OMS／リスク（段階導入 §11）
- 暗号：**bitbank** APIで現物発注、スリッページ/手数料モデル、約定照合（D1）。
- 日本株：証券API（IBKR/国内）。空売りコスト・規制を制約に組込。
- リスク：ポジション/グロス/レバ限度、ドローダウン連動デリスク、**キルスイッチ**、構造変化検知で戦略停止 [KB §8-3]。

### L13 監視
- live-vs-backtest Sharpe減衰、スリッページ実測vsモデル、容量（キャパシティ）、DSR時系列、実効ポジション数（getEffNum）。

## 11. 段階的ロールアウト（フェーズ・ゲート）

| Phase | 内容 | 前進ゲート（定量） |
|-------|------|--------------------|
| **0 研究基盤** | L1-L8構築（取込〜検証ハーネス・試行レジストリ）。取引なし | パイプライン再現性100%、CPCV/DSRが既知データで正しく動作 |

## 12. リスク管理
- **サイジング**＝メタラベル確率からベットサイズ（過信を抑制）[KB §4.2]。
- ハード限度：1ポジション上限、グロス/ネット・エクスポージャ、レバ上限。
- ドローダウン連動デリスク＋**キルスイッチ**。
- **構造変化/レジーム検知**で当該戦略を自動停止 [KB §8-3]。
- **再校正**＝月次再fit＋週次監視＋レジーム検知のイベント駆動（D3）。頻度はライブ前に事前登録し、成績を見て反応的に変えない。
- 意図ポジションと約定の**照合**（執行不全時の残余リスクをヘッジ）。
- **共和分の崩壊監視（柱D）**：共和分は構造変化で壊れる（恒久でない）。ライブのスプレッドを継続再検定し、崩壊を構造変化検知 [KB §8-3] と連動して自動デリスク [KB §11.6]。
